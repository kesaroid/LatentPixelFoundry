# V2 Architecture -- Future Roadmap

This document describes the planned V2 architecture. None of this is implemented in V1.

---

## V2 Architecture Overview

```
User --> Dashboard --> Backend API --> Redis Queue --> GPU Worker Pool
                          |                               |
                          |                               v
                          |                         Object Storage (S3)
                          |                               |
                          v                               v
                     PostgreSQL                    Instagram API
```

In V2, the system shifts from a synchronous trigger model to an asynchronous queue-based pipeline with external storage and social media integration.

---

## Planned Enhancements

### 1. Object Storage (S3-Compatible)
- Replace local `generated_videos/` directory with S3-compatible object storage (AWS S3, MinIO, Cloudflare R2)
- Worker uploads directly to object storage after generation
- Backend stores a public URL in the `video_path` column instead of a local file path
- Eliminates disk space constraints on the backend server
- Enables CDN distribution for faster downloads

### 2. Redis Job Queue
- Replace direct HTTP trigger with a Redis-based job queue (e.g., `arq`, `celery`, `bull`)
- Backend enqueues jobs; workers dequeue and process
- Enables natural backpressure and job prioritization
- Decouples backend from worker availability

### 3. Automatic Instagram Posting
- After video reaches COMPLETED, optionally post to Instagram via Graph API
- Add `instagram_status` and `instagram_post_id` columns to Job table
- Configurable per-job: user can choose to auto-post or just download
- Requires Instagram Business Account and Facebook App registration

### 4. User Authentication
- Add user accounts with email/password or OAuth (Google, GitHub)
- Associate jobs with users
- Rate limiting per user
- JWT-based session management

### 5. Concurrency Control
- Limit concurrent generation jobs per user
- Global worker pool capacity management
- Queue depth monitoring and alerts

### 6. Autoscaling Workers
- Deploy workers on autoscaling GPU infrastructure (RunPod, Modal, AWS Batch)
- Scale to zero when idle to minimize cost
- Scale up based on queue depth
- Health check and automatic worker replacement

### 7. Job Retry Mechanism
- Configurable retry count per job (default: 3)
- Exponential backoff between retries
- Dead letter queue for permanently failed jobs
- Retry only on transient errors (timeout, network), not on invalid prompts

### 8. Analytics Dashboard
- Job success/failure rates over time
- Average generation time by resolution/duration
- Queue depth and worker utilization
- Cost tracking per video generated

### 9. Prompt Enhancement
- Optional GPT-based prompt expansion before sending to video model
- Prompt history and favorites
- Prompt templates and presets

### 10. Multi-Model Support
- Support multiple video generation models (different quality/speed tradeoffs)
- Model selection per job
- A/B testing between models

## V2 Database Schema Changes

```sql
-- New columns on jobs table
ALTER TABLE jobs ADD COLUMN user_id UUID REFERENCES users(id);
ALTER TABLE jobs ADD COLUMN storage_url TEXT;
ALTER TABLE jobs ADD COLUMN instagram_status VARCHAR(20);
ALTER TABLE jobs ADD COLUMN instagram_post_id TEXT;
ALTER TABLE jobs ADD COLUMN retry_count INTEGER DEFAULT 0;
ALTER TABLE jobs ADD COLUMN max_retries INTEGER DEFAULT 3;
ALTER TABLE jobs ADD COLUMN model_name VARCHAR(100);

-- New tables
CREATE TABLE users (
    id UUID PRIMARY KEY,
    email TEXT UNIQUE NOT NULL,
    hashed_password TEXT NOT NULL,
    created_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE prompt_templates (
    id UUID PRIMARY KEY,
    user_id UUID REFERENCES users(id),
    name TEXT NOT NULL,
    template TEXT NOT NULL,
    created_at TIMESTAMPTZ DEFAULT now()
);
```

## Migration Path from V1

1. **Storage migration:** Write a script to upload existing `generated_videos/` files to S3 and update `video_path` to URLs
2. **Queue migration:** Deploy Redis, update backend to enqueue instead of HTTP trigger, update worker to dequeue
3. **Auth migration:** Add users table, create admin account, associate existing jobs with admin
4. **Zero downtime:** Each enhancement can be deployed independently behind feature flags
